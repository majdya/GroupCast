Overview

GroupCast is a real-time group chat application for LAN environments. It uses TCP for all management operations (registration, login, group management) and UDP multicast for actual chat message delivery.

Built for a university Data Communication course, it demonstrates practical networking concepts including socket programming, protocol design, and multicast communication.

TCP Management

Client-server communication for registration, login, and group lifecycle management.

UDP Multicast

Efficient group message delivery using IP multicast — one send reaches all group members.

TLV Protocol

Custom binary protocol (b,b,B) for all management messages: 1B Type, 1B Length, 255B max payload.

Architecture

CLIENT SERVER ────── ────── Screen 1 (Register/Login/Exit) select() loop Screen 2 (Create/Join/Leave/Logout) ├── Comm_Accept() │ ├── Comm_TryRecv() → dispatch: ▼ ├── ServerMng (User Hash) ClientMng │ register / login / logout │ └── GroupMng (Group Hash) ▼ create / join / leave / free MC IP Comm-Link (TLV framing) │ ▼ TCP socket

The server uses a single-threaded select() event loop — no threads, no locks. The Comm-Link layer abstracts TCP framing behind a clean API, keeping business logic (ServerMng, GroupMng) network-agnostic.

Full end-to-end

# Terminal 1 — start server
./server

# Terminal 2 — start client (register → login → create group → chat)
./client

The client shows two menu screens:

  • Screen 1: Register, Login, Exit
  • Screen 2: Create Group, Join Group, Leave Group, Logout

On group creation or join, the server returns a multicast address + port. The client then spawns chat_sender and chat_receiver in gnome-terminal windows for chat. (Requires gnome-terminal installed.)

pkt.length = header[1];

    receive_response(pkt.payload, pkt.length);

Demo Flow

The complete end-to-end user journey:

1. Register

Client sends REGISTER_REQ with username + password. Server validates uniqueness, stores user.

2. Login

Client sends LOGIN_REQ. Server verifies credentials, marks user active.

3. Create Group

Client sends CREATE_GROUP_REQ. Server checks name uniqueness, allocates multicast address.

4. Join Group

Client sends JOIN_GROUP_REQ. Server returns multicast addr + port. Client spawns chat windows.

5. Chat

chat_send reads stdin, sends to UDP multicast. chat_recv listens on multicast, prints to stdout.

6. Leave Group

Client sends LEAVE_GROUP_REQ. Server decrements member count, may auto-close group.

7. Logout

Client sends LOGOUT_REQ. Server leaves all groups, marks inactive. Client closes all chat windows.
